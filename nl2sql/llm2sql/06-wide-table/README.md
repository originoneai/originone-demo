# 06 · 宽表 / 指标表 / 三查询面对照（配套 C1-LLM-05A / 05B）

对应课程文章：《把四张表压成一张宽表：你替模型先把路走一遍，它就不容易走错》《回看：宽表这条路，本质是拿数据工程的确定性去换模型即兴的风险》。

前几课模型在跨表题上要么撞死、要么绕圈，病根都是**表太多、关系太绕**。这一课换个打法：
用数据工程，在**建表期**就把该 join 的接好、该固化的口径写进注释，让模型运行时看到的
从"四张要自己拼的源表"，变成"一张查询友好表"。搜索空间被压小，C1 直出 SQL 更稳。

```
物理表面   4 张源表 → 模型自己选表、接 join、扛粒度（搜索空间最大）
宽表面     order_item_wide（明细粒度）→ join 和类目已接好，但扇出坑还在
指标表面   order_daily_metric（日粒度预聚合）→ 趋势题过滤日期即可，几乎零出错面
```

## 文件

| 文件 | 作用 |
|---|---|
| `cli.py` | **交互终端（主入口）**：`\physical`/`\wide`/`\metric` 切面，`\ab <题>` 物理vs宽表并排，`\truth` 基准答案 |
| `views.py` | 建 `order_item_wide`（明细粒度宽表，口径写进注释）+ `order_daily_metric`（日指标） |
| `surfaces.py` | 三查询面的 schema prompt + `run_on_surface` + 类目销售额基准答案（抓扇出双算） |
| `db.py` / `guard.py` | 连接层 / 只读门禁 |
| `build_dataset.py` | 生成 SQLite 样例库（订单 1~3 条明细，天然带扇出） |
| `test_lab.py` | 视图建成 + 扇出成立 + 三面 Prompt 递减 + 基准自洽 + 扇出坑成立 + DeepSeek 真实对比 |

## 快速开始

```bash
pip install -r requirements.txt
export DEEPSEEK_API_KEY=你的key
python cli.py
```

先在宽表面问「每个类目的销售额是多少？」，再敲 `\ab 每个类目下，订单实付金额合计是多少？`，
你会亲眼看到扇出双算——在明细粒度宽表上直接 `SUM(order_actual_amount)`，金额会翻好几倍
（`\truth` 对一下真值就露馅）。

## 这一课要你看见的两件事

1. **宽表帮的不只是"少接 join"，更是把口径知识前置成模型看得见的上下文。** `order_item_wide`
   的两行注释——"订单级金额会重复、别直接 SUM""明细级金额可安全求和"——就是它的灵魂。
   模型看懂注释后，用 `item_amount` 算销售额、用 `COUNT(DISTINCT order_id)` 数订单，都对了。
2. **宽表不是魔法：粒度选明细，就得背扇出这个包袱。** 你保住了下钻能力，代价是订单级字段
   在明细行上会重复，直接聚合就翻倍。这坑靠"把口径写进注释 + 查询时回收粒度"来兜，
   靠"以为建好就万事大吉"来栽。

## 和真实企业库的关系

这个 4 张表的小库，看不出宽表压缩上下文/join 的震撼——真正的威力要到几百张表、
一道题物理 Prompt 近八千字的真实库上才显形（宽表可能压到四成、指标表一成）。这里先把
机制吃透：换库只改 `DB_URL`，`views.py` 的 `CREATE VIEW` 换成你自己的 join 即可。
